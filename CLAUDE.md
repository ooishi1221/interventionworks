# 組織マニュアル (Intervention Works Edition)

## ミッション

"Intervene in the world. Rebel against their values."

AIとプロダクトを融合させ、既存の価値観に介入することで、ライダーの自由と新たな事業価値を最速で実装する。

## 「誰の価値観だ？」 — 意思決定の原則

機能を足す前に **「誰の価値観だ？」** と問え。それが我々のライダーから出た声でないなら、他のアプリの模倣でしかない。

世の中にあふれる「正解」という名の厚い壁。それらは本当にライダーが求めたものなのか。常識という名の平穏に対する反乱 — それが Intervention Works の全プロダクトに通底する思想である。

### プロダクト一覧

Intervention Works は複数のプロダクトを同時に推進する。思想のOSは共通、プロダクトごとにフォルダを分離し、作業切り替え時は `/clear` する。

### Moto-Logos のコンセプト: 存在証明

Moto-Logos は「バイク駐車場の検索アプリ」ではない。**都市ライダーの存在証明**である。

ライダーの体験すべて — 停めた場所、食った飯、見た景色、走った道 — をワンショット（写真1枚）で刻む。AIが写真を判別し、自分のノートに残り、誰かの地図になる。ライダーは分類しない。撮るだけ。

バイク駐車場はその最初の検証カテゴリ。都市ライダーが日常的に直面する「停める場所がない」痛みで、ワンショットの仕組みが機能するかを証明する。

- **目的:** 俺はここにいた、を刻む
- **手段:** ワンショット（写真1枚）で足跡を刻む。AIが分類。足跡の集積が誰かの地図になる
- **最初の検証:** バイク駐車場（都市特有の痛み。地方ツーリングでは困らない）
- **ターゲット:** 首都圏の都市ライダー。拡張は都市間横展開（東京→大阪→名古屋→福岡）
- **世界観:** 評価しない、競わない、ただ「いた」ことが刻まれる

> 「ワンショットが足跡を刻む。足跡が誰かの地図になる。」

### 実例: Moto-Logos での判断

| 「常識」 | 問い | 我々の答え |
|---------|------|-----------|
| スポット登録はフォームで入力 | 誰の価値観だ？ | グローブしたまま写真1枚で終わるべき。残りはみんなで育てる |
| レビューは星評価 | 誰の価値観だ？ | 駐車場は評価するものじゃない。情報が合ってたかどうかだ |
| ユーザーの貢献はランクで可視化 | 誰の価値観だ？ | 存在に序列はない。ランクもポイントも不要。足跡の数だけ見せろ |
| ユーザーに「報告」を求める | 誰の価値観だ？ | プラットフォームの都合。ライダーは自分のためにメモするだけ。それが勝手に誰かの役に立つ |
| UGCの善意に頼る | 誰の価値観だ？ | 善意の搾取。到着後に「自分のため」にアプリを開く理由を設計しろ |

### Slight のコンセプト: Zero-Resistance

Slight は「電熱グローブ」ではない。**ライダーとバイクの間にある抵抗をゼロにする**ブランドである。

バイク業界は「機能」を言い訳に「美学」を捨て、「安全」の名のもとに野暮ったい装備を押し付けてきた。Slight はその常識に介入する。

- **思想:** "Whose values?" / "Don't be a jerk." / DISTURB NOTHING
- **目的:** 存在を感じさせないほど馴染むギアで、ライダーの触覚と美学を取り戻す
- **3フェーズ:**
  - Phase 1: Tactile Integrity（触覚の復権） — 0.5mm極薄、指先CNTヒーター
  - Phase 2: Aesthetic Liberation（美学の解放） — 街に溶け込むシルエット
  - Phase 3: Cultural Insurrection（文化的な反乱） — 「安全ならダサくていい」への反論

> "EXIT THE ORDINARY."

**D2Cモデル実証としての位置づけ:**
Slightの第一弾（電熱インナーグローブ）は、企画→ファブレス製造→CF販売を一気通貫で完遂する実証プロジェクト（PoC）でもある。利益の最大化ではなく、プロセスの完遂証明が目標。成功プロセスをパッケージ化し、ウィットワンのD2C伴走コンサルの営業エビデンスにする。

- **価格:** 5,000円（テストマーケティング特化）
- **ロット:** 300個（CF受注生産、在庫リスクゼロ）
- **No-Go基準:** CF開始14日で目標50%未達なら凍結
- **投資額:** 約30万円（サンプル費・認証着手金）

### 実例: Slight での判断

| 「常識」 | 問い | 我々の答え |
|---------|------|-----------|
| 電熱グローブは分厚くて当然 | 誰の価値観だ？ | 0.5mmで指先だけ温める。操作性を殺すなら本末転倒 |
| 安全のためにゴツくていい | 誰の価値観だ？ | 街に溶け込むシルエット。ダサさは安全の代償じゃない |
| 温度調整は3段階ボタン | 誰の価値観だ？ | 物理スイッチなし。繋げばON、外せばOFF。グローブしたまま操作させるな |
| USBケーブル同梱 | 誰の価値観だ？ | 全員持ってる。同梱はコストと環境の無駄 |
| パッケージは箱 | 誰の価値観だ？ | 車検証入れになる耐水ケース。捨てられないパッケージにしろ |

## Security

- `.env.local`, `serviceAccount*.json` は**絶対にcommitしない**
- APIキー・パスワードをコード・設定ファイルにハードコードしない
- Firebase Admin SDK の鍵はローカル環境のみ（`/tmp/` or 環境変数経由）
- permissions（settings.local.json）にクレデンシャルを含めない

### 6. ライブラリ更新ポリシー

- **Expo SDK の minor 更新**: `npx expo install --check` で推奨版に合わせる
- **Expo SDK の major 更新** (例: 54→55): `npx expo upgrade` で一括。必ず別ブランチで
- **ネイティブ依存の個別更新**: `npx expo install <pkg>@latest` で最新取得 → 型チェック → EAS preview で試す
  - newArch 絡みで `--check` が古い版を推奨するケースあり（例: react-native-maps 1.20.1 推奨 → 1.27.2 が必要）
- **更新後は必ず preview ビルドで実機検証**。JS だけの変更なら OTA でも OK、ネイティブ依存は要フルビルド

## 作業ルール

- **作り始める前に確認せよ** — 不明な点があれば `AskUserQuestion` で選択肢付きの質問をしてからCEOの判断を仰ぐ。推測で進めない。
- **検証してから完了を報告せよ** — 実装後は必ず型チェック・ビルド・dev server起動等で動作を確認する。UI変更はスクリーンショットを撮って目視確認。「tscが通った」だけでは検証にならない。
- **タスクが変わったら `/clear` を促せ** — Admin→LP→アプリのように異なるプロジェクトの作業に移る場合、CEOに「`/clear` してから次に行きますか？」と確認する。コンテキスト汚染を防ぐ。
- **大きなタスクはPlan Modeから始めよ** — 3ファイル以上を変更する実装タスクは、まず Plan Mode（Ctrl+G）で探索・計画してからコーディングに入る。
- **調査にはサブエージェントを使え** — コードベースの調査はサブエージェントに委譲し、メインコンテキストを軽く保つ。
- **デプロイはスキルを使え** — `/deploy-admin`, `/deploy-lp` で標準化されたデプロイフローを実行する。
- **Issueベースの作業は `/fix-issue` を使え** — `/fix-issue 123` でIssue取得→調査→実装→検証→コミット→クローズの一連フローを実行する。

## 実機バグ対応プレイブック（深掘り仮説に入る前の必修チェック）

実機で「動かない」「表示されない」「クラッシュ」系の報告を受けたとき、**仮説ドリブンで深掘りに走る前に以下を機械的に確認**する。これを飛ばすと 2026-04-20 の `EXPO_PUBLIC_*` 未注入で数時間ハマった事例を再発する。

### 最初の30秒でやる3点
1. **環境変数の注入確認**: `cd engineering/moto-logos && npm run preflight [preview|production]`
   - EAS Secrets と `.env` の差分 / app.json 必須項目 / Firestore rules を一括チェック
2. **Sentry 管理画面**: https://moto-logos-team.sentry.io/projects/moto-logos/ で直近Issueを確認
   - 既に Slack `#moto-logos-dev-log` にリアルタイム通知が飛ぶので、そこも即確認
3. **CEOにデバッグシェア依頼**: 設定 → サポート → 「デバッグ情報を開発者に送信」を押してもらう
   - Firestore `debug_reports` に書き込まれ、Slack Bot が整形通知（uid / build / update ID / 直近エラー3件）

### 深掘り仮説に入る判断軸
- 上記3点で **手がかりが見つからない場合のみ** 仮説検証に入る
- 1つの仮説に固執しない。修正を3回重ねても再現するなら**前提を疑い直す**（API Key の存在確認、ビルド内容の実機目視など）
- 症状のヒアリングは「何が起きないか」より「どこまでは起きるか」を優先（診断Alertで段階切り分け）
